# US-23 — TLS-Only Connections

**As a** security-conscious user,
**I want** my email client to NEVER send passwords or data without TLS encryption,
**so that** my credentials and mail content are always protected in transit.

---

## Acceptance criteria

| # | Criterion |
|---|-----------|
| 1 | `imap://` in `EMAIL_HOST` is rejected at startup with a clear error message directing the user to use `imaps://`. |
| 2 | `smtp://` in `SMTP_HOST` is rejected at startup with a clear error message directing the user to use `smtps://`. |
| 3 | At connection time, any `imap://` URL is refused even if somehow passed directly to the IMAP layer. |
| 4 | The log always shows the TLS version in use (e.g. `TLS/TLSv1.3`) upon successful connection. |
| 5 | When a connection is attempted without TLS, a `WARN` log entry is written. This code path is only reachable via `SSL_NO_VERIFY=1` in test environments and must not be reachable in production. |
| 6 | All test mock servers (IMAP and SMTP) use TLS with self-signed certificates generated at build time by CMake. |

---

## Error message format

When an insecure URL is detected in config:

```
Error: Insecure IMAP host rejected: imap://example.com
       Use imaps:// to enforce TLS-encrypted connections.
```

```
Error: Insecure SMTP host rejected: smtp://example.com
       Use smtps:// to enforce TLS-encrypted connections.
```

---

## Logging format

On successful TLS handshake:

```
[INFO] TLS connection established: TLSv1.3 / TLS_AES_256_GCM_SHA384
```

On insecure connection (test environments only, `SSL_NO_VERIFY=1` must be set):

```
[WARN] Connecting without TLS — this is only permitted in test environments
```

---

## Implementation notes

* Validation is performed in `config_load_from_store()` or immediately before the
  first network call, before any credentials are transmitted.
* The `curl_adapter` layer logs the TLS version from `CURLINFO_TLS_SESSION` after
  a successful connection.
* The `SSL_NO_VERIFY=1` bypass is intentionally limited to development/test use;
  it must not suppress the WARN log entry.
* Mock servers in `tests/functional/` and `tests/pty/` already use TLS (certs
  generated by the `test-tls-certs` CMake custom target).
